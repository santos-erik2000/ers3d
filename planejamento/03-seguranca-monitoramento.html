Documento de planejamento — não é código

Como o sistema se protege — e como sabemos se algo quebrou

Etapa 3: aprofunda a estratégia de segurança e monitoramento sobre a arquitetura já decidida na Etapa 1 (Next.js único, Auth.js, Neon, R2, Render) — sem repetir o que já foi coberto lá.

01 Autenticação & sessão

  • E-mail + senha via Auth.js (credentials provider) — sem SSO/OAuth no MVP, conforme decidido na Etapa 1.
  • Hash de senha com argon2 (mais resistente a força bruta que bcrypt para hardware atual).
  • Sessão em cookie httpOnly + Secure + SameSite=Lax, expiração deslizante razoável (ex.: 8h de inatividade) — sem necessidade de refresh token separado nessa escala.
  • Rate limiting no endpoint de login (ex.: 5 tentativas / 15 min por IP+e-mail) para mitigar força bruta.
  • Bloqueio/inativação de usuário pelo Admin em vez de exclusão — usuário bloqueado não consegue autenticar, mas seu histórico de autoria permanece íntegro na auditoria.

02 Proteções da aplicação

AmeaçaMitigação
SQL InjectionPrisma parametriza toda query — nenhuma query SQL crua concatenada com input do usuário
XSSReact escapa output por padrão; nenhum dangerouslySetInnerHTML com conteúdo de usuário; cabeçalho Content-Security-Policy restritivo
CSRFCookie SameSite=Lax + verificação de origem nos Route Handlers/Server Actions que alteram estado
Entrada inválidaTodo boundary de entrada (formulário, API) validado com Zod antes de chegar à camada de domínio
Permissão só no front-endToda ação sensível é checada de novo no service de domínio no backend — o front-end reflete o resultado, nunca decide sozinho (ver Etapa 1, riscos técnicos)
Upload maliciosoValidação de extensão + MIME type real do arquivo (não só a extensão do nome); limite de tamanho; nome de arquivo gerado pelo sistema, nunca o nome original salvo como path

03 Segredos & configuração

Nenhum segredo (senha de banco, chave de API, secret de sessão) no código-fonte ou versionado no git. Variáveis de ambiente configuradas diretamente no painel do Render por ambiente (dev/homologação/produção), com valores distintos entre eles — a chave de sessão de produção nunca é reaproveitada em homologação.

04 Logs de segurança & auditoria

Complementa o interceptor de audit_logs já modelado na Etapa 1. Eventos de segurança específicos a registrar, além das operações de negócio:

  • Login bem-sucedido e falho (com IP/contexto quando disponível)
  • Bloqueio/desbloqueio de usuário
  • Mudança de permissão de um usuário
  • Exportação de dados (relatórios, financeiro)
  • Correção administrativa de registro com justificativa (ROOT/Admin)

05 Backup & recuperação

  • Neon mantém backups automáticos com point-in-time recovery no free tier — validar o período de retenção coberto e documentar.
  • R2 (arquivos): habilitar versionamento de objeto para proteger contra sobrescrita/exclusão acidental de anexos.
  • Teste de restore deve fazer parte do checklist antes do primeiro lançamento em produção — um backup nunca testado é uma suposição, não uma garantia.
  • Nenhuma migration destrutiva roda em produção sem esse backup confirmado antes (já registrado como regra na Etapa 1).

06 LGPD

Pendência conhecida

Não existe hoje uma política de privacidade/texto de consentimento (confirmado na Etapa 1). O que segue é a infraestrutura técnica que o sistema já pode oferecer — o conteúdo jurídico do texto de consentimento continua sendo uma tarefa fora do desenvolvimento, a ser resolvida antes do lançamento em produção.

  • Consentimento: campo de aceite + timestamp no cadastro do cliente, pronto para referenciar o texto jurídico assim que existir.
  • Exportação de dados pessoais: endpoint/ação que reúne os dados de um cliente específico (cadastro, interações, pedidos) — infraestrutura simples de implementar, dado que a página 360° do cliente já centraliza essa mesma informação.
  • Correção: já coberta pelo CRUD normal de cliente com auditoria.
  • Retenção e exclusão: exclusão lógica por padrão (Etapa 1) — para uma eventual solicitação de exclusão real, definir período de retenção mínimo (obrigações fiscais/contratuais) antes de anonimizar de fato.

07 Monitoramento & operação

Erros

Sentry (free tier, 5k eventos/mês) capturando exceções não tratadas no servidor e no cliente, com contexto de usuário/rota.

Disponibilidade

Endpoint /health simples (verifica conexão com banco) — usado pelo próprio Render para decidir se a instância está saudável.

Ambientes

Local (Docker/dev) → Homologação (Render, branch staging) → Produção (Render, branch main), com bancos Neon separados por ambiente.

Rollback

Redeploy do commit anterior no Render (histórico de deploys nativo da plataforma) — sem infraestrutura extra necessária nessa escala.

08 Checklist pré-lançamento

  • Backup do banco confirmado e restore testado ao menos uma vez
  • Variáveis de ambiente de produção revisadas (nenhum segredo de dev reaproveitado)
  • Rate limiting de login ativo
  • Todas as rotas sensíveis validadas com teste de permissão negada (não só permissão concedida)
  • CSP e cabeçalhos de segurança configurados
  • Texto de consentimento LGPD definido e vinculado ao campo já existente no cadastro
  • Sentry recebendo eventos de teste em produção antes do primeiro uso real
  • Endpoint /health respondendo e monitorado
